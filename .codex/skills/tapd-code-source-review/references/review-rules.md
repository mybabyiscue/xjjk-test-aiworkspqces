# 初步代码审查规则

## 审查目标

初步审查只用于发现明显阻塞项和高风险信号，决定代码源是否适合进入后续 `tapd-testcase-code-review`。

本阶段不做测试用例与代码实现的深度关联分析。

## 审查维度

必须关注：

- 硬编码 token、password、secret、host、access key。
- 可疑敏感日志输出。
- TODO、FIXME、临时代码、调试代码。
- 明显危险 SQL 或字符串拼接 SQL。
- 外部调用缺少异常处理的迹象。
- 明显空值风险。
- 权限校验疑似缺失。
- 状态流转缺少保护。
- 事务边界明显缺失。
- 重复提交或幂等保护缺失的明显迹象。

## 问题分级

- P0：严重安全、数据破坏或阻塞后续分析的问题。
- P1：高风险缺陷，需要开发确认或修复。
- P2：可疑问题、代码质量问题或建议优化。

## 输出要求

`code_prepare_findings.md` 必须包含：

- 问题等级。
- 服务编号。
- 文件路径。
- 行号。
- 问题描述。
- 风险影响。
- 建议处理方式。

`raw/prepare_findings.json` 必须保留结构化问题清单，供后续复查或汇总读取。

## 敏感信息处理

不得输出完整密钥、token、password、cookie、authorization header。发现疑似敏感值时，只输出字段名、文件位置和脱敏摘要。
