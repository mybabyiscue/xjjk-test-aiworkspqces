# 执行接口测试并生成测试报告 (test-execute-from-tapd)

## 1. 目标与范围

本技能读取 `output/interface_test_preparation.md` 中定义的用例测试准备详情及参数结构，以及 `output/integration_test_flow.md` 的集成测试脚本骨架，直连测试数据库，在测试环境运行接口验证，最终输出通过率和执行报告。

准备手册是人工/上游流程写的 Markdown，不是保证干净的结构化数据。手册里经常出现变量占位符不是合法 JSON、变量没登记来源、URL 或骨架脚本里混着字面值等问题。本技能**不假设手册是对的**，执行前必须先做一遍解析和一致性校验，把有问题的用例挑出来单独处理，而不是带着错误数据默默跑过去。

## 2. 前置输入

- **配置参数**：`environments_config.json`
- **接口测试手册**：`output/interface_test_preparation.md`
- **集成流脚本骨架**：`output/integration_test_flow.md`

以上文件缺失，或按第 3.1 节的方式解析后一条可执行用例都拿不出来，直接停止并说明具体原因，不生成执行报告。

## 3. 执行要求

### 3.1 解析手册，摸清用例总量和变量来源

先完整解析 `interface_test_preparation.md`，按接口分节和参数表提取全部用例（Case ID、接口路径、方法、正向/反向属性、参数映射表、请求体、预期响应）。用例总数以这次解析结果为准，不写死具体条数。

解析时占位符不是合法 JSON，不能直接 `json.loads()` 请求体原文，必须先把 `{{ ... }}` 里的表达式单独提取出来求值，再拼回请求体。占位符统一按以下规则识别：

- `DB.<表名>.<字段名>`：来自"测试数据准备"章节登记的反查结果。
- `ENV.<路径>`：来自 `environments_config.json`。
- `上一步 [步骤N] 返回的 <字段路径>`：来自集成流程中前一步 of 真实响应。
- `INVALID_<NAME>`：反向用例的构造值，占位符里可能带括号注释（例如 `INVALID_ID (e.g. 999999)`），解析时只取 `INVALID_ID` 作为变量名，括号内容当成人读提示，不参与匹配，但要在报告里保留原文方便核对。

解析完成后做一次自检，产出"解析与一致性检查"清单，包含以下几类问题，任何一类命中都不能直接放行执行，要单独列出来：

- **格式解析失败**：参数表、请求体缺失或格式不符合预期，无法提取出完整用例。
- **变量来源缺失**：请求体或参数表里出现的变量（`DB.*`、`ENV.*`、`INVALID_*`），在"测试数据准备"章节或 `environments_config.json` 里找不到对应的登记条目。这种变量不能凭表名字面意思猜一个查询方式去补，必须原样标记为缺失。
- **文档内部矛盾**：同一用例里，参数映射表/请求体用变量表示某参数，但 URL 或请求体其他位置对同一参数写了具体字面值（比如参数表说 `id` 来自 `{{ DB.special_column.id }}`，但请求 URL 却写死 `?id=1`）。这种情况以参数映射表为准还是以字面值为准无法自动判断，必须标记为"待确认"，不能默认选一个跑。
- **骨架脚本与参数表不一致**：`integration_test_flow.md` 或手册第 5 节的示例脚本里，某个变量的取数表（查哪张表、哪个字段）和该用例在第 4 节参数映射表里登记的来源不一致。骨架脚本只是执行辅助参考，不是真值来源；出现不一致时，以逐用例的参数映射表为准，并在报告里注明骨架脚本需要同步修正。
- **重复覆盖**：多个 Case ID 解析出的接口、方法、请求体完全相同（哪怕用例标题描述的业务场景不同）。这种情况实际执行时只发一次真实请求，避免重复调用测试环境，但报告里要把合并的 Case ID 全部列出来，并标注"业务场景未通过参数区分，需要人工补充差异化数据"，不能按用例数分别计入通过率把问题掩盖掉。

只有以上五类问题清单都确认过（问题条目已知晓且不阻塞执行，或用户已明确同意跳过），才进入 3.3 的实际执行。有问题但未处理的用例不执行、不计入通过率，只出现在报告的"待处理"部分。

### 3.2 测试驱动构建

- 在 `scratch/test_runner.py` 生成执行脚本，脚本里禁止硬编码任何主键、Token 或域名，这些一律从 `environments_config.json` 或数据库动态查询获取。
- 变量解析逻辑按 3.1 节的规则统一实现为一个独立的占位符解析函数，不要为每个用例各写一段特例代码；查询用的 SQL 以手册"测试数据准备"章节登记的语句为准，不要自己另编查询条件。
- 必须包含从数据库动态提取可用主键和答案的逻辑，取数方式（连接、查询条件）需要能在报告里追溯到手册里对应的登记条目。

### 3.3 正反向用例遍历执行

- 遍历 3.1 解析且通过一致性检查 of 用例（正向 + 反向），逐条执行，不预设跑多少条，也不因为某类用例数量少就跳过。
- 3.1 标记为"重复覆盖"的一组用例只发一次真实请求，其余按合并处理，结果同步应用到组内所有 Case ID。
- 为每个变体传入手册中对齐好的实际 Payload（正常值/异常值），发送真实网络请求。

### 3.4 集成流程执行（独立于单接口用例）

`integration_test_flow.md` 里的核心业务链路和 `interface_test_preparation.md` 里的单接口用例是两件不同的事：单接口用例互相独立、允许某一条失败不影响其他；集成流程是一条链，按步骤顺序执行，前一步失败或拿不到需要传递的响应值，后面的步骤就不该继续跑。

- 按 `integration_test_flow.md` 里 mermaid 图和"API 调用详情"章节定义的顺序，逐步执行，不跳步、不并发打乱顺序。
- 串联调用时必须抓取上一步的真实响应值并传给下一步，防止空指针或参数缺失。例如把领奖接口 `/live/reward/receive` 返回的 `id` 传入发奖回填接口 `/live/reward/rewardsuccess`。这里的接口名是举例，实际以 `integration_test_flow.md` 里定义的流程为准；如果集成流程骨架脚本里的字段取数表和手册里对该步骤的参数说明不一致，按 3.1 节的"骨架脚本与参数表不一致"规则处理，不擅自二选一。
- 某一步执行异常、断言失败，或者需要传给下一步的字段没能从响应里取到，立即中断整条流程，标记中断发生在第几步、原因是什么，后续未执行的步骤原样标记为"未执行"，不跳过继续硬跑、也不假装后面几步是通过的。
- 一份手册如果定义了多条核心流程（比如正常路径 + 异常中断路径），逐条按上面的规则独立执行，互不影响。

### 3.5 断言与验证

- 断言 HTTP 状态码是否为预期状态。
- 断言响应 JSON 体中的 `code` 和 `msg` 是否符合用例的正向成功或反向拦截要求。
- 必要时断言物理数据库状态（数据是否落库、字段值是否符合预期）。

### 3.6 Token 失效与交互式回填 (Token Expiration & Interactive Backfilling)

测试在调用接口时，由于测试环境 Token 具有时效性，极易出现会话失效。执行引擎必须能够识别 Token 过期，并支持中断、交互式输入以及自动重试，确保测试流的自愈。

#### 3.6.1 失效检测逻辑 (Expiration Recognition)
执行引擎必须在每次接口请求后，对 HTTP 状态码及响应体 JSON 进行如下校验：
- **状态码校验**：如果 HTTP 状态码为 `401` 或 `403`。
- **错误码/信息校验**：如果响应 JSON 的业务错误码（如 `code` 或 `Code`）属于 Token 失效相关代码（如 `A00002`、`2003`），或者响应文案中包含 `Token已失效`、`Token失效`、`登录已过期`、`未授权` 等特征。
- **多租户隔离降级**：如果接口对于任何请求（甚至查询接口）均统一返回 `A00001 资源不存在`，亦应视为 Token 或 Session 失效，触发回填检测。

#### 3.6.2 交互中断协议 (Halt & Prompt Protocol)
一旦判定 Token 失效，执行引擎应立即触发以下中断和提示动作：
1. **终端交互（交互式环境）**：
   - 在终端打印明显的 `[TOKEN_EXPIRED_ERROR]` 旗标及所属环境名称。
   - 调用命令行输入（如 Python `input()`），提示用户：“检测到环境【环境名】的 Token 已失效，请输入新的 Token 回填（或 Enter 跳过并退出）：”。
   - 若用户输入了新 Token，自动将新 Token 回填并写入 `environments_config.json` 的对应环境项下，返回成功标记。
2. **静默退出（非交互式/后台任务）**：
   - 若在非交互式终端中捕获到 `EOFError`（如 IDE 自动化执行后台任务），引擎必须打印 `[TOKEN_EXPIRED_ERROR] <环境名>` 并**以特定的退出码（例如 `Exit Code 10`）立即退出**。
   - AI 代理（Agent）捕获到该退出码后，必须**在聊天对话框中直接暂停，向用户索要新 Token**。收到后，代理使用写文件工具将 Token 更新至 `environments_config.json`。

#### 3.6.3 原地更新与自动重试 (Update & Inline Retry)
- **原地更新**：在回填新 Token 后，执行引擎应在内存中**重新加载最新的 `environments_config.json` 变量**，并用新 Token 动态替换掉后续请求（及当前失败请求）的 `Authorization` 头部信息。
- **自动重试**：引擎对当前导致中断的接口直接发起**原地重试（最大重试次数为 2 次）**。如果重试成功，测试继续向下执行，整个测试流无需从头全部重新运行，保障了测试的连续性与效率。

### 3.7 租户自适应判定与决策 (Tenant Auto-detection & Fallback Decision)

由于 SaaS 环境包含多租户物理隔离，执行引擎在配合多租户平台进行数据校验时，必须执行自动权重分析与交互决策机制，并且向下完美兼容非租户（单租户）平台。

#### 3.7.1 自动加权判定 (Auto-detection)
- 执行引擎连库后，如果配置文件未指定租户 ID，应统计核心表中 `tenant_id` 的分布行数并进行加权求和。
- 若单一租户的加权行数总占比超过预设阈值（如 70%），引擎自动采用该租户 ID，无需用户干预。

#### 3.7.2 决策唤醒与代理交互 (Ambiguity Gate)
- 当租户 ID 出现多个候选且权重接近、或无法确定时，引擎需打印 `[TENANT_AMBIGUITY_ERROR]` 并输出候选租户列表。
- 在交互式终端下，允许用户输入选择的租户 ID；在后台静默执行模式下，抛出 **`Exit Code 11`** 立即中止。AI 代理捕获该状态码后，需唤醒界面选择组件（`ask_question`），引导用户选择租户，回填配置文件并重试。

#### 3.7.3 单租户平台向后兼容 (Single-Tenant Backward Compatibility)
- 执行引擎必须在连库时动态探测目标数据表是否包含 `tenant_id` 字段（例如使用 `SHOW COLUMNS FROM <table> LIKE 'tenant_id'`）。
- 若目标表不包含该字段，引擎必须无条件自动降级为“单租户模式”：
  1. 屏蔽并移除所有 SQL 语句中的 `tenant_id` 过滤逻辑，改行全表反查。
  2. 自动过滤掉 API 请求参数表和 Payload 中的 `tenantId` 字段（将其设为空或直接从 payload 结构中移除）。
  3. 绕过任何与租户选择相关的弹窗决策，保障非多租户系统的平滑验证。


#### 3.8 测试断言失败的自愈与深层数据诊断 (Assertion Failure & Data Self-Healing)

在测试执行过程中，若接口返回业务层错误码（如 A00001、A00002），执行引擎与 AI 代理严禁直接判定用例失败并向用户挂起汇报。必须启动“数据自愈与排查”机制：

##### 3.8.1 自愈与重试协议 (Self-Healing Protocol)
1. **活动/资源失效自愈**：
   - 若报错原因指示“活动不存在”、“抽奖不存在”等状态性阻断，引擎必须主动通过 SQL 扫描相关活动表（如 `lottery_activity`），查找当前是否存在状态为“进行中/有效”（如 `status = 1`）的可用记录。
   - 发现有效数据后，自动更新内存中的 ID 变量，并对该接口执行原地重试（最多 2 次）。
2. **多端/双端关联绑定自愈**：
   - 若接口返回“参数非法”或“未绑定客户/用户”，引擎必须去物理数据库中查找具有联合映射关联的记录（如 `wework_customer` 表中同时不为 NULL 的 `unionid` 和 `customer_id`）。
   - 提取出真实处于绑定状态的联合数据作为 Payload 重新回填并发送。

##### 3.8.2 报告与汇报原则
- 只有在数据库中彻底找不到任何有效活动或绑定用户，或接口报 5xx 物理故障时，方可判定用例执行失败，并在报告中详述。
- 自动重试成功后，应在报告中备注“已通过数据自愈自动修复（替换为有效关联数据）”，确保测试流最大化通过。


## 4. 产出交付物

测试完成后在 `output/` 目录下生成两份报告，分别对应 3.3 的单接口用例和 3.4 的集成流程——两者验证的对象和失败语义不一样（单接口是一条条独立的，流程是一条链、一步断全断），不要合并成一份笼统报告。

### 4.1 《单接口测试执行报告》（`output/interface_test_execution_report.md`）

- 测试执行概览：通过率、环境配置、用例总数（取自 3.1 的解析结果）、待处理用例数量。
- **解析与一致性检查清单**（放在报告最前面）：按 3.1 节五类问题分别列出涉及的 Case ID 和具体原因，这部分内容不计入通过率，但必须让人一眼看到有哪些用例没跑、为什么没跑。
- 明细表格：Case ID、API 路径、断言结果、用例属性（正向/反向），重复覆盖的用例注明"与 Case X 合并执行"。
- 所有失败用例的请求/响应参数快照与失败原因解析。

### 4.2 《核心流程测试报告》（`output/core_flow_test_execution_report.md`）

- 流程概览：流程名称、涉及的接口数量、本次是否端到端跑通（闭环成功 / 中途中断）。
- 按步骤顺序的明细：每一步的接口（Method/URL）、实际发送的请求体（占位符解析后的真实值）、响应快照、断言结果、该步骤是否成功把需要的字段传给了下一步（具体传了哪个字段、值是什么）。
- 中断记录（如果发生）：在第几步中断、原因、后续未执行的步骤列表——不能因为流程中断就不生成报告，中断本身就是重要信息。
- 涉及的真实数据来源：这条流程用到的数据库查询记录，方便回溯这次跑的是哪批真实数据。
- 如果手册里定义了多条流程，每条流程单独一节，不要混在一起统计通过率。

## 5. 失败处理

- 手册解析不完整、变量来源缺失、文档内部矛盾、骨架脚本与参数表不一致、用例重复覆盖：这五类都算"待处理"，在报告开头单独列出，不静默跳过、不擅自选择一种取值去跑、不假装它们跑过了。
- 请求执行异常（网络、鉴权失效等）：如实记录该用例为"执行异常"，区别于"断言失败"和"待处理"，三者不要混在一起算通过率。
- 集成流程中途中断：不算作某一步"失败"就完事，整条流程标记为"未闭环"，在《核心流程测试报告》里写清楚中断步骤和原因，后续步骤原样标记"未执行"，不要因为流程断了就跳过不生成报告。
